Hello! The following parameters and specifications apply to comp-driven website (re)design projects, to ensure a smooth transition to front-end development and building CMS templates. My goal is to outline parameters to consider how designs will behave when they’re translated into code, or interacted with. These parameters and specifications will also help put a11y (accessibility) first, rather than retrofitting a11y to a design. (Learn more about designing for a11y.)

  1. All comps must include 3 responsive states, covering all designed elements and states: Mobile, Tablet, Laptop/Desktop.
  2. If regions contain full-bleed content that has a max-width, such as large hero photos, the comp’s width should exceed that width, to show what happens beyond the design element’s max-width dimensions.
  3. All typography treatments should be documented in a styleguide, including font-family, font-weight, color, and font-size. Any single-use deviations from those styles must be annotated in comps.
  4. Styleguide must include specs for:
    • Paragraph / Heading 4
    • Unordered list
    • Ordered list
    • Blockquote
    • Heading 1
    • Heading 2
    • Heading 3
  5. ems are multiples of a font’s size. line-height, margins, padding, and height can all be coded in ems so that if the user changes their default browser font-size, the content styles will scale. While not required, it can be very helpful to think in these terms, and line-height and margin are typically coded using ems. (The line-height of this text, for example, is 1.5em. The paragraph margins are also 1.5em, so that they equal 1 blank line.)
  6. Icons should be produced and provided as SVGs. Bitmapped UI and brand elements should be provided as (minimum) @2x sliced PNG-24s with transparent backgrounds.
  7. Production files should be delivered as flattened PNGs, @2x, for reference and grabbing dimensions and offsets. (Your developer neither needs to keep up with versions of Photoshop or Sketch or the latest and greatest design tool, nor spend time clicking in and out of design files to fetch values that should comprise a styleguide.)
  8. All form fields must include :focus and non-:focus states.
  9. Form field designs that deviate from browser defaults must include annotations noting that such designs will require likely JavaScript coding to render consistently, particularly with styled select menus (dropdowns) or labels inside text input fields. Sign-off on this approach must be secured.
  10. (ANY styling of select fields (dropdowns) rendered consistently across browsers will require a dependency on Harvest’s Chosen JavaScript plugin and be achieved within its constraints.)
  11. (Labels really shouldn’t appear inside text input fields. HTML5 included the placeholder attribute for a reason, to provide an example of appropriate input and formatting. The label should be outside the input element, to allow proper use of the placeholder attribute. In-field labels also disappear on users once they’ve entered a value, which leads to ambiguity or obfuscation as they fill in field values.)
  12. All comps must include a version where any optional content is removed, to show how elements otherwise flow together.
  13. Vertical margins between layout elements should be identified and must precede 1 element, not be a combination of adjacent element spacing.
  14. All dimensions, gutters, and offsets should be in even-numbered increments. Responsive CSS is math. Even numbers divide evenly.
  15. No single-line text elements should be included in horizontal lock-ups, as text *will* wrap and break those layouts.
  16. Text shapes on the same designed element should not vary in width on a page-by-page basis to control where text wraps. If this is a goal, it must be annotated in comps and sign-off secured.
  17. Any scroll effects, hover effects, or other behaviors must be annotated at each breakpoint.
  18.   19. We’re embracing the CMS or we’re coding static pages. We cannot code in content-specific breakpoints or CSS rules on a page-specific or section-specific basis for the sake of navigation or other elements on a site that will have editable, configurable content. CMS-based pages have titles which are output in navigation and links. Those need to be designed flexibly.
  20. Requests for clarification on consistency or intent incur time to discuss the issue at the expense of maximizing productive development time.
  21. Lastly, remember that no matter how straightforward the design, the content model, or how amenable or savvy the client, design comps are an aspiration, to be tempered by a practical CSS design system and the cascade, browser constraints, CMS constraints, or other variables. They are not a promise.
