--- title: Front-end Web Development Specifications --- {% include head.html %}

Hello! The following parameters and specifications apply to comp-driven website (re)design projects, to ensure a smooth transition to front-end development and building CMS templates. My goal is to outline parameters to consider how designs will behave when they’re translated into code, or interacted with. These parameters and specifications will also help put A11y (accessibility) first, rather than trying to make a design accessible.

All comps must include 3 responsive states, covering all designed elements and states: Mobile, Tablet, Laptop/Desktop.

If regions contain full-bleed content that has a max-width, such as large hero photos, the comp's width should exceed that width, to show what happens beyond the design element's max-width dimensions.

All typography treatments should be documented in a styleguide, including font-family, font-weight, color, and font-size. Any single-use deviations from those styles must be annotated in comps.

Styleguide must include specs for:

  • Paragraph / Heading 4
  • Unordered list
  • Ordered list
  • Blockquote
  • Heading 1
  • Heading 2
  • Heading 3

Icons should be produced and provided as SVGs. Bitmapped UI and brand elements should be provided as (minimum) @2x sliced PNG-24s with transparent backgrounds.

Production files should be delivered as flattened PNGs, @2x, for reference and grabbing dimensions and offsets.

All form fields must include :focus and non-:focus states.

Form field designs that deviate from browser defaults must include annotations noting that such designs will require likely JavaScript coding to render consistently, particularly with styled select menus (dropdowns) or labels inside text input fields. Sign-off on this approach must be secured.

(ANY styling of select fields (dropdowns) rendered consistently across browsers will require a dependency on Harvest’s Chosen JavaScript plugin and be achieved within its constraints.)

(Labels really shouldn’t appear inside text input fields. HTML5 included the placeholder attribute for a reason, to provide an example of appropriate input and formatting. The label should be outside the input element, to allow proper use of the placeholder attribute. In-field labels also disappear on users once they’ve entered a value, which leads to ambiguity or obfuscation as they fill in field values.)

All comps must include a version where any optional content is removed, to show how elements otherwise flow together.

Vertical margins between layout elements should be identified and must precede 1 element, not be a combination of adjacent element spacing.

All dimensions, gutters, and offsets should be in even-numbered increments. Responsive CSS is math. Even numbers divide evenly.

No single-line text elements should be included in horizontal lock-ups, as text *will* wrap and break those layouts.

Text shapes on the same designed element should not vary in width on a page-by-page basis to control where text wraps. If this is a goal, it must be annotated in comps and sign-off secured.

Any scroll effects, hover effects, or other behaviors must be annotated at each breakpoint.

Content revealed in hover states is a bad idea – The Microsoft Surface notebooks include a detachable screen that can be used in tablet mode, but would match laptop/desktop breakpoints per viewport width; Surface also supports touch events when in laptop mode. Any goals of hiding and revealing content will require extra work for touchscreen detection, and must be annotated in comps and signed off on.

Requests for clarification on consistency or intent incur time to discuss the issue at the expense of maximizing productive development time.
